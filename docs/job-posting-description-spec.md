# Googleしごと検索 description 拡充 仕様書（ドラフト）

> 目的：`tools/job-generator.html` で生成する求人ページの JobPosting 構造化データのうち、
> `description` プロパティを「meta description の使い回し（120文字）」から、
> Googleしごと検索の詳細パネルで実際に読まれる**長文の求人票**に作り替える。
>
> 最終更新：2026-07-20（初版・調査結果ベースのドラフト）／担当：豊永
>
> **このドキュメントの位置づけ**：本チャットではリサーチと仕様策定のみ行い、実装は別チャットで行う。
> 実装担当は本ファイルと [job-html-template.js](../assets/js/job-html-template.js)・[job-generator.html](../tools/job-generator.html) を読んだ上で着手すること。

---

## 0. 背景・きっかけ

- `tools/job-generator.html` は求人ページ（`recruit/job-*.html`）とその JobPosting 構造化データ（JSON-LD）を生成する社内ツール。
- ラベルの色分け（黒＝Googleしごと検索に渡る項目、灰＝自社サイト表示のみ）を導入する過程で、`description` プロパティが「meta description（検索結果スニペット用、120文字推奨）」をそのまま流用していることが判明。
- Google公式ドキュメント（[Search Central: Job posting](https://developers.google.com/search/docs/appearance/structured-data/job-posting)）によれば `description` に文字数制限はなく、"complete job details including responsibilities, qualifications, and requirements" を入れることが期待されている。`<br>` `<p>` `<ul>` `<li>` タグが認識される。
- 実際に運営者がスマホでGoogleしごと検索を確認したところ、他社は詳細パネルにかなり長い本文を掲載しており、一定量まではそのまま表示され、それ以降は「全文表示」ボタンで展開される挙動だった。
- → 実例を調査した上で、`description` の構成方針を決める。

### 1-2. 構造化データが確認できなかった／取得できなかったサイト（12件）

| サイト | 状況 |

| エン転職 | ページ取得可だが ld+json 自体が存在しない |
| Green | 取得可だが schema.org 系の JSON-LD なし |
| Wantedly | ld+json はあるが Organization / WebSite 型のみ、JobPosting なし |
| 保育士バンク! | `FAQPage` / `BreadcrumbList` のみで JobPosting なし |
| マイナビ保育士 | `BreadcrumbList` のみ |
| LITALICOキャリア | `BreadcrumbList` のみ（複数求人ページで確認） |
| キララサポート保育 | `BreadcrumbList` のみ |
| スターバックス（自社採用） | `BreadcrumbList` のみ。そもそも個別求人ページを持たない構成 |
| ヤマト運輸（自社採用） | ld+json 自体が存在しない |
| doda | ネットワークが数回リトライしても全てタイムアウトし取得不能（ボット対策の可能性） |
| リクナビNEXT | 静的HTMLは取得できたが中身がほぼ空（15KB）。JavaScriptによるSPA描画のため求人内容が取得できず |

→ **共通点**：**「まず登録・エージェント相談」を前提とするマッチング型サービス**か、**JavaScriptでの動的レンダリングのため（サーバー返却HTMLに）構造化データが載っていないサイト**が大半。JobPostingを実装していたのは、いずれも**直接応募・検索完結型の求人ボード／老舗転職メディア**だった。Google for Jobs対応は媒体によって温度差が大きい。

### 1-3. 共通パターンのまとめ

1. **文字数**：3,500〜6,500字程度。meta description（120字）とは桁が2つ違う「求人票の完全版」。同業種（放課後等デイ・保育・児童指導員）のジョブメドレー／ウェルミージョブ／レバウェルでも3,500〜5,200字あった。
2. **タグの使い方**：`<br>` が主体。`<p>` `<ul><li>` を使うのはマイナビ転職のみ、ウェルミージョブに至っては**タグを一切使わずプレーンテキスト＋改行だけ**でも成立している。→ ハピネスキッズも `<br>` 中心のシンプル構成で十分。
3. **見出しスタイル**：「■見出し名」を罫線（`----` や `━━━`）で挟むスタイルが多数派（4/5サイト）。ウェルミージョブのみ装飾なしの素の見出し行。
4. **構成順序**（5サイトを平均した典型的な並び）：
   1. 求人の要点／ポイント（マイナビ・レバウェルが冒頭に配置）
   2. 募集職種・雇用形態・応募資格などの基本条件
   3. 仕事内容
   4. 職場の特徴・アピール
   5. 給与（本給＋昇給・賞与＋諸手当に分けることも）
   6. 勤務時間（勤務体制を分けることも）
   7. 休日・休暇（長期休暇を分けることも）
   8. 福利厚生・待遇
   9. 教育体制・研修制度
   10. 施設情報（施設名・形態・定員・スタッフ構成・設立年月日・設備）・アクセス
   11. スタッフの声／取材レポート／メッセージ／選考プロセス／応募方法
5. **「仕事内容を上→処遇（給与・勤務条件）を中→企業アピール・施設情報を下」**という「実務→処遇→企業アピール」の大きな流れは全サイト共通。ユーザーが実機（スマホ）で確認した「上に一定量表示され、それ以降は全文表示ボタンで展開」という挙動とも整合し、**一番読ませたい仕事内容・魅力を前半に置く**設計が理にかなっている。

## 2. 現状の実装（変更前）

[job-html-template.js](../assets/js/job-html-template.js) の `generateHTML(v)` 内：

```js
const metaDesc = v.metaDesc || `高知市の放課後等デイサービス ハピネスキッズで${v.title}を募集しています。`;
...
const jsonLdObj = {
  ...
  "title": v.title, "description": metaDesc,
  ...
};
```

`description` に `metaDesc`（120文字推奨のSEOスニペット用テキスト）をそのまま流用している。フォームには仕事内容・応募資格・歓迎条件・勤務時間・休日休暇・給与補足などが入力済みだが、これらは自社ページのHTML本文にしか反映されておらず、Googleしごと検索側には渡っていない。

## 3. 提案：description と meta description の分離

| 用途 | 使う場所 | 文字数目安 | 中身 |
|---|---|---|---|
| `<meta name="description">` | 検索結果スニペット（切れたら終わり、スクロール不可） | 120文字前後（現状維持） | 既存の「meta description」欄をそのまま使用（変更不要） |
| JobPosting `description` | Googleしごと検索の詳細パネル（長文OK、展開ボタンあり） | 制限なし（既存フィールドを組み立てた結果、自然に1,000〜3,000字程度になる見込み） | 仕事内容・応募資格・歓迎条件・勤務時間・休日休暇・給与・給与補足・勤務地・応募方法を見出し付きでまとめた長文 |

**方針**：新しい入力欄は増やさず、既存フォームの入力内容（仕事内容・応募資格・歓迎条件・勤務時間・休日休暇・給与補足など）を自動的に組み立てて `description` を生成する。

### 3-1. 見出しスタイル

このツールは既に `renderHw()`（ハローワーク用テキスト出力）で `【見出し】` 形式を使っている（[job-generator.html](../tools/job-generator.html) の `renderHw()` 参照）。調査した他社は「■見出し＋罫線」が主流だったが、**社内の既存の記法と統一するため `【見出し】` 形式を踏襲する**ことを提案する（罫線スタイルへの変更は必須ではない・好みで良い）。

### 3-2. 提案する構成順序とセクション（案）

```
【仕事内容】
（jobContent）

【応募資格】
（requirements）

【歓迎条件】
（preferred）※空なら省略

【給与】
月給／時給 ○○円 〜 ○○円
（salaryNote）※空なら省略

【勤務時間】
（hours）※空なら省略

【休日・休暇】
（holidays）※空なら省略

【勤務地】
〒780-0931 高知県高知市玉水町5 岩貞ビル
放課後等デイサービス ハピネスキッズ

【応募方法】
メール：mail@happiness-kids.net
電話：088-824-8810（担当：豊永）
```

- 各セクションの本文は改行を `<br>` に変換して埋め込む（= 既存の `nl2br()` を流用）。
- 値が空のセクションは丸ごと省略する（現状の `prefRow` 等と同じ「空なら出さない」方針を踏襲）。
- 先頭に `v.intro`（ページ冒頭の紹介文）を「求人の要点」的な位置づけで置くかどうかは実装時に判断（他社の「求人のポイント」枠に相当）。

### 3-3. 実装イメージ（job-html-template.js）

```js
function buildJobPostingDescription(v) {
  const ul = v.salaryType === "MONTH" ? "月給" : "時給";
  const fmt = n => (n || 0).toLocaleString("ja-JP");
  const sections = [];
  if (v.intro)        sections.push(`【求人の紹介】\n${v.intro}`);
  if (v.jobContent)   sections.push(`【仕事内容】\n${v.jobContent}`);
  if (v.requirements) sections.push(`【応募資格】\n${v.requirements}`);
  if (v.preferred)    sections.push(`【歓迎条件】\n${v.preferred}`);
  sections.push(`【給与】\n${ul} ${fmt(v.salaryMin)}円 〜 ${fmt(v.salaryMax)}円${v.salaryNote ? "\n" + v.salaryNote : ""}`);
  if (v.hours)         sections.push(`【勤務時間】\n${v.hours}`);
  if (v.holidays)      sections.push(`【休日・休暇】\n${v.holidays}`);
  sections.push(`【勤務地】\n〒780-0931 高知県高知市玉水町5 岩貞ビル\n放課後等デイサービス ハピネスキッズ`);
  sections.push(`【応募方法】\nメール：mail@happiness-kids.net\n電話：088-824-8810（担当：豊永）`);
  return sections.map(nl2br).join("<br><br>");
}
```

そして `jsonLdObj` 側は：

```js
"description": buildJobPostingDescription(v),
```

`<meta name="description">` タグ側（HTML `<head>`）は従来どおり `metaDesc`（短いSEOスニペット）を使い続ける。**変更しない**。

## 4. 影響範囲・注意点

- 変更対象は [assets/js/job-html-template.js](../assets/js/job-html-template.js) の `generateHTML()` のみ（ブラウザの `tools/job-generator.html` と Node.js の `scripts/regenerate-pages.js` の両方に自動反映される共通実装）。
- `tools/job-generator.html` 側のフォーム項目・入力欄は変更不要（既存フィールドを組み立てるだけなので新規入力欄は増えない）。
- 学歴・経験要件（`educationRequirements` / `experienceRequirements`、2026-07-20実装済み）は、descriptionとは別のプロパティなので今回の変更対象外。ただし将来的に「description内にも学歴・経験要件を文章で触れるか」は検討の余地あり（優先度低）。
- description が長くなることで、既存の `recruit-detail-*` ページのHTML本文（人間向け表示）には影響しない（JSON-LDは`<script type="application/ld+json">`内なので非表示要素）。
- 求人ごとに入力量が異なるため、実際の文字数は300字程度（最小限しか入力していない求人）から2,000字超（全欄埋めている求人）まで幅が出る想定。Google側の上限がないため問題ない。

## 5. 調査で見つかった「追加コンテンツ案」（採否は要検討）

調査した他社が `description` に載せていた要素のうち、**現状のハピネスキッズのフォームには無いが、追加すると訴求力が上がりそうなもの**を列挙する。いずれも実装するには「フォームに新しい入力欄を足す」か「テンプレートに固定文を埋め込む」必要があり、今回の「既存項目の組み立てだけ」の範囲を超える。次のチャットで採否を判断する材料として記録する。

| 追加候補 | どのサイトにあったか | ハピネスキッズでの活かし方 | 実装コスト |
|---|---|---|---|
| **ハッシュタグ**（#未経験可 #ブランクOK #子育て中の方歓迎 等） | ジョブメドレー | 冒頭 or 末尾に検索性・目を引く効果。既存の「歓迎条件」から半自動生成も可 | 低（固定 or フォーム追加） |
| **スタッフの声／先輩職員の声** | ウェルミージョブ | 現場の雰囲気を伝え共感訴求。定型でも効果あり | 中（フォーム欄追加） |
| **採用担当者からのメッセージ** | ウェルミージョブ | 「担当：豊永」から一言メッセージ。応募ハードルを下げる | 中（フォーム欄追加） |
| **職場の特徴・取材レポート的な紹介** | レバウェル／ジョブメドレー | 施設の雰囲気・支援方針を物語的に。既存 `intro` の拡張でも可 | 低〜中 |
| **教育体制・研修制度** | ジョブメドレー | 未経験・ブランク層に安心材料。放課後等デイは未経験入職も多い | 中（フォーム欄追加） |
| **選考プロセス**（応募→面接→採用の流れ） | ジョブメドレー | 応募者の不安解消。定型文で埋め込み可能 | 低（固定文） |
| **施設情報**（施設形態・利用者定員・スタッフ構成・設立年月日・設備） | ジョブメドレー／レバウェル | 事業所の規模感を伝える。ほぼ固定値なのでテンプレート埋め込み向き | 低（固定文） |
| **「この仕事に向いている人／向いていない人」** | マイナビ転職 | ミスマッチ防止。障がい児支援は適性が重要なので相性良い | 中（フォーム欄追加） |
| **昇給・賞与・諸手当を給与とは別セクションに分割** | マイナビ転職／ジョブメドレー | 現状は `salaryNote` 1本。細分化すると見やすい | 低（既存項目の分割表示） |
| **想定年収** | ジョブメドレー | 月給×12＋賞与の目安。求職者が最も気にする数値 | 低（自動計算 or フォーム追加） |

> 注：これらは「あると良い」候補であり、全部入れると入力負担が大きくなる。**まずは第3章の「既存項目の組み立てだけ」で最小実装し、効果を見てから施設情報・選考プロセス・スタッフの声あたりを段階的に足す**のが現実的、というのが本調査時点での推奨。

## 7. 実装記録（2026-07-20 実装完了）

第3〜6章の検討を経て、運営者と合意した「案3-改」で実装した。ここに最終仕様を記録する。

### 7-1. 採用したアーキテクチャ（案3-改）

情報を3層に分け、**同じデータから「JobPosting description（Googleしごと検索）」と「表示ページHTML」の2出力**を生成する（単一ソース・2出力）。セクションの順序と内容は [job-html-template.js](../assets/js/job-html-template.js) の `buildSections()` が唯一の定義点。

| 層 | 中身 | 管理場所 |
|---|---|---|
| L1 求人別・構造化 | 職種・給与・仕事内容・応募資格 ほか | jobs.json（求人フォーム） |
| L2 共通プロフィール | 会社名・住所・電話・メール・担当者・施設形態・定員・対象・スタッフ構成・設立・営業日・送迎、および職場の特徴/福利厚生/研修/選考フローの共通初期値 | **`assets/data/profile.json`（新設）** |
| L3 求人別・任意フリー | 求人のポイント・ハッシュタグ・こんな方に向いています・スタッフの声・採用担当者メッセージ | jobs.json（求人フォーム） |

- **共通プロフィール（L2）**：従来テンプレートにハードコードしていた会社名・住所・電話等もすべて `profile.json` へ移設。ツールに「共通プロフィール」タブを新設し、そこで編集→GitHubへアップロード。`regenerate-pages.js`（GitHub Actions）と `tools/job-generator.html`（ブラウザ）の両方が読む。
- **共通項目の上書き**：職場の特徴・福利厚生・研修・選考フローは、各求人でチェックを入れると個別に上書き可（`<key>Override`）。各欄に「共通に戻す」ボタンあり。上書きが空ならプロフィールの値を使う（`effective()`）。

### 7-2. 出力順序（確定）

`【求人のポイント】`（＋特徴・歓迎キーワード）→ 仕事内容 → 職場の特徴・支援方針 → 応募資格 → 歓迎条件 → こんな方に向いています → 給与（基本給／昇給・賞与／諸手当）→ 勤務時間 → 勤務体制 → 休日・休暇 → 長期休暇 → 福利厚生・待遇 → 教育・研修体制 → スタッフの声 → 施設情報 → 選考の流れ → 採用担当者からのメッセージ → 勤務地・アクセス → 応募方法。**空／未チェックの項目は自動省略**。

**設計意図**：給与・勤務地・雇用形態はGoogleが詳細パネル上部の別枠（構造化チップ）で先に表示するため、description本文の冒頭は「ポイント→仕事内容→職場の魅力」で惹きつけに使う。応募可否（応募資格・歓迎）を早めに示して離脱を防ぎ、処遇→施設情報・選考フロー・担当者メッセージで信頼を固めて応募へ。放課後等デイ＝未経験・ブランク・子育て層が多い前提で、研修体制・向き不向き・スタッフの声を安心材料として配置。

### 7-3. 第6章の未確定事項の決定

1. 見出しスタイル → **`【】` を踏襲**（社内renderHw記法と統一。ただしrenderHw自体は7-5で廃止）。
2. `intro` → 表示ページのhero（冒頭）に置く。descriptionでは、職場の特徴（appeal）が空のとき `intro` をフォールバックとして「職場の特徴・支援方針」に使う。
3. セクション区切り → **`<br><br>`（空行1つ）**。
4. 学歴・経験要件のdescription本文への文章化 → 見送り（従来どおり `educationRequirements`/`experienceRequirements` プロパティのみ）。
5. 既存ページ → **`regenerate-pages.js` で全件再生成済み**。
6. 第5章の追加コンテンツ → 運営者判断で**フル実装**（施設情報・選考フロー・研修・スタッフの声・向いている人・採用担当者メッセージ・給与の昇給賞与/諸手当分割 まで全て。想定年収は不採用）。各項目はチェックボックスで求人ごとに出す/出さないを選択。
   - **ハッシュタグは不採用**：静的サイトでは`#`が検索・絞り込みとして機能せず、企業HPの品位の観点からも自然文が適切と判断。「特徴・歓迎キーワード」欄として残し、`#`を付けず「未経験可・ブランクOK」のように**中黒区切りの自然文**で表示する（pillにしない）。内部データキーは `keywords`。

### 7-4. profile.json の初期値

`about/index.html` から拾えた値を投入済み（施設形態・定員10名・対象・営業日・送迎・スタッフ構成・住所・会社名・連絡先）。**未取得の6項目（設立年月日・職員の具体人数・研修体制の中身・選考フロー・福利厚生・賞与実績）は空欄**。運営者が後から共通プロフィールタブで埋める（空欄なら出力に出ない）。`hiringOrganization.name` は法人名「合同会社足人」。

### 7-5. ハローワーク出力タブの廃止

運営者の指示により、`tools/job-generator.html` の「ハローワーク」タブ（ハローワーク入力用テキスト生成 `renderHw()` 等）は**仕組みごと削除**。「共通プロフィール」タブに置き換えた。※求人ページ側の「ハローワーク求人番号」入力と応募案内表示は別機能として存続。

### 7-6. 残タスク（別途）

- **ツールの設置場所**：`tools/job-generator.html` が公開サイト上にありPATをlocalStorageに持つ問題（CLAUDE.md既知懸念）。公開サイトから切り出す方針は合意済みだが、設置先（NAS／非公開リポジトリ）は保留。切り出す際は、共有テンプレート・profile.json を絶対URLで読むよう変更する。
- 出力順序・可視ページのレイアウトは後から調整可能（`buildSections()` の配列と `generateHTML()` のHTML組み立てを編集）。
